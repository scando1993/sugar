1

Phase 1 — Planning

Analyze the task and produce a complete execution plan. No code.
1

Receive & Analyze Task

Read the user's $ARGUMENTS. Identify scope, complexity, and potential dependencies. Ask clarifying questions if the task is ambiguous.

2

Create plan.md

Write a comprehensive execution plan at the repo root containing:

  • Objective — clear, measurable goal
  • Scope — what's included and excluded
  • Assumptions & Constraints
  • Risks — potential failure points
  • Dependency map — which parts depend on others
  • Architecture strategy
  • Execution phases with dependency annotations
3

Create todo.md

Break the plan into small, actionable, idempotent tasks grouped by phase. Each task gets a checkbox (- [ ]), dependency annotations, and completion criteria.

4

Stop & Await Approval

Present plan.md and todo.md to the user. Do not proceed until explicitly approved. No code, no branches, no implementation.

Decision: User Approval

Approved — Proceed to Phase 2: Workspace Setup
Rejected — Revise plan.md and todo.md based on feedback
Prompt repetition: "The original task is: $ARGUMENTS. This phase produces plan.md and todo.md only."
2

Phase 2 — Workspace Setup

Create isolated git worktrees and branches for each execution phase.
1

Create Phase Directories

Create the base directory at /tmp/<repo>-phases/ to hold all phase workspaces.

2

Create Git Worktrees

For each phase in plan.md, run git worktree add with a new branch:

  • phase-a-types/ on branch phase-a-types
  • phase-b-api/ on branch phase-b-api
  • phase-c-ui/ on branch phase-c-ui

Each workspace is a full copy of the repo with its own branch. Changes don't cross-contaminate.

3

Initialize progress.txt

Create a progress log in each workspace with a header and timestamp. This file will accumulate learnings and codebase patterns during implementation.

4

Stop & Await Approval

Confirm all worktrees and branches are set up correctly. Document any setup issues in plan.md.

Decision: User Approval

Approved — Proceed to Phase 3a: Dependency Analysis
Rejected — Fix workspace issues, re-verify setup
3a

Phase 3a — Dependency Analysis

Map what each phase produces, consumes, and depends on.
1

Read plan.md

Thoroughly review the execution plan to understand each phase's scope and intended outputs.

2

Map Phase Relationships

For each phase, determine:

  • Produces — artifacts, files, APIs, types, schema
  • Consumes — what it needs from other phases
  • Hard deps — must complete before this starts
  • Soft deps — helpful but not blocking
  • Shared files — cross-phase modifications
3

Build Dependency Graph

Map all produces/consumes relationships. Check for circular dependencies (halt if found). Identify the critical path — the longest sequential chain that determines minimum total time.

4

Identify Execution Groups

  • Group 1 — phases with zero dependencies (run first, in parallel)
  • Group 2 — phases that depend only on Group 1
  • Sequential chains — linear dependencies that must run in order

Decision: Circular Dependency?

No cycles — Proceed to Phase 3b: PRD Generation
Cycle found — Halt. Requires manual resolution before continuing
3b

Phase 3b — PRD Generation

Create execution documentation and Ralph workspace files for every phase.
1

Create execution.md

Write the execution plan at repo root with:

  • ASCII dependency graph
  • Parallel execution groups (numbered)
  • Execution order with satisfied dependencies
  • Critical path analysis
  • Risk assessment (conflict likelihood, shared files)
2

Generate prd.json per Phase

Convert each phase's tasks into Ralph-format user stories. Each story must be:

  • Completable in one agent pass
  • Describable in 2-3 sentences
  • Ordered by internal dependency
  • Include "Typecheck passes" as final criterion
  • All start with passes: false
3

Generate CLAUDE.md per Phase

Agent instructions for each workspace. Contains the full Ralph protocol: read prd.json, pick story, implement, quality check, commit, update state, stop. Includes prompt repetition — task restated at top and bottom.

4

Generate ralph-loop.sh per Phase

The iteration engine. A bash loop that spawns fresh agent instances — one per story. Detects PHASE_COMPLETE to exit. Made executable with chmod +x.

5

Generate VERIFY.md per Phase (Consensus Mode)

When prd.json has a consensus config, generate verifier agent instructions with iron laws, rationalization table, and VOTE:PASS/VOTE:FAIL output format. Only created for consensus-mode phases.

6

Generate patterns.json Schema

Define the patterns.json schema for structured pattern propagation between execution groups. Includes pattern name, description, applies_to scope, and example.

After this step: Each phase workspace has 6 files — prd.json, progress.txt, CLAUDE.md, ralph-loop.sh, VERIFY.md, and patterns.json (between groups) — ready for autonomous execution.

Phase 3c — Parallel Execution

Launch Ralph loops for all independent phases simultaneously.

Parallel Execution Model

Group 1
phase-a-types
ralph-loop.sh
phase-b-api
ralph-loop.sh
phase-c-ui
ralph-loop.sh
Group 2
phase-d-integration
ralph-loop.sh (with patterns from G1)
phase-e-tests
ralph-loop.sh (with patterns from G1)
1

Launch Group 1 (No Dependencies)

All phases with zero dependencies launch sugar run <workspace> as its own separately-backgrounded call — not multiple processes joined with a single foreground wait, which would exceed typical tool execution time limits. Each loop runs independently.

2

Wait for Group Completion

Wait for all Group 1 processes to exit. Check exit codes — 0 means all stories passed, non-zero means max iterations reached or error.

3

Propagate Codebase Patterns

Collect Codebase Patterns sections from all completed phases' progress.txt files. Inject these learnings into the next group's CLAUDE.md context.

4

Launch Next Group

Start Group 2 phases (which depend on Group 1). Repeat pattern: launch in parallel, wait, propagate patterns. Continue until all groups complete.

5

Sync Tracking Files

Update todo.md checkboxes and execution.md with actual results. sugar run does not push branches automatically — pushing is a deliberate, separate step the user or Phase 4 takes explicitly.

Decision: All Stories Passing?

All pass — Proceed to Phase 4: Merge
Some blocked — Check prd.json notes, fix blockers, resume with sugar run <workspace>
4

Phase 4 — Merge

Safely integrate all completed phase branches in dependency order.
1

Create merge_order.md

Document the recommended merge sequence, informed by the dependency graph:

  • Merge order (foundations first)
  • Rationale for ordering
  • Expected conflict areas
  • Conflict resolution strategies
  • Validation steps after each merge
2

Merge in Order

Merge each phase branch into main in the documented order. Foundation phases first, dependent phases after. One merge per commit.

3

Validate After Each Merge

Run full quality checks after each merge: typecheck, lint, tests. If any fail, fix before proceeding to next merge. Document actual conflicts in merge_order.md.

4

Final Validation

Run the complete test suite on the fully merged result. Phase 4 is not complete until all checks pass on the final merged code.

Decision: Final Validation

All pass — Workflow complete. All phases merged and validated.
Failures — Fix issues, update merge_order.md, revalidate
R

The Ralph Loop

One story per agent, fresh context each iteration, memory via files.
The Ralph pattern is the core execution engine. Each phase workspace runs this loop independently via sugar run <workspace> — a TypeScript command (LoopRunner) that spawns fresh agent instances one per story, with model tiering (auto-escalation on repeated failures, persisted across restarts) and consensus verification (Raft-inspired quorum voting via Verifier/sugar verify). ralph-loop.sh still exists per workspace but is now a one-line wrapper that execs sugar run.

Iron Laws

1

ONE STORY

Implement exactly one user story per iteration, then stop. No "while I'm here" additions.

✗

NEVER COMMIT BROKEN

Every commit must pass all quality checks. No exceptions, no skipping.

📖

READ PROGRESS.TXT FIRST

Always read codebase patterns before writing a single line of code.

Model Tiering

○

Default Model

Start with a cost-effective model (e.g., Sonnet). Configurable per phase via sugar run <workspace> --max-iterations 20 --model sonnet.

↑

Auto-Escalation

After 2 consecutive failures (ESCALATION_THRESHOLD), upgrade to a more capable model (e.g., Opus). No manual intervention needed.

↓

De-Escalation

After a success on an escalated model, drop back to the default. Keeps cost efficient while handling hard stories.

Memory Persistence (across iterations)

{ }

prd.json

State machine. Which stories are done (status: "passed"), which are blocked (status: "blocked" + notes field). LoopRunner reads this to pick and claim the next story.

𝒫

progress.txt

Learnings, patterns, and gotchas. The Codebase Patterns section at the top is read first — ensuring each new agent benefits from prior discoveries.

⌥

git history

Every completed story is an atomic commit. Code persists across iterations via the branch. The new agent starts from where the last one left off.

Error recovery: Before each attempt, a snapshot tag namespaced by phase and story (sugar/<phase>/<story-id>/attempt-<n>) is created for rollback — namespacing avoids collisions since git tags are repo-global and parallel phases would otherwise share the same tag name. If quality checks fail 3 times, the loop writes a structured report to failure_log.json (storyId, attempt, filesModified, failureType, lastError), resets changes, and moves to the next story. Future agents read failure_log.json to try a DIFFERENT approach.
✦

Critical Concepts

The principles and patterns that make the workflow reliable at scale.
📐

Plan Before Code

Phase 1 always runs first. The entire execution plan, dependency map, and task breakdown are completed and approved before any code is written or branches are created.

🔒

Isolated Workspaces

Each phase gets its own git worktree and branch. Changes in one phase never affect another until the explicit merge in Phase 4. Enables true parallel execution.

1

One Story at a Time

The Ralph pattern: each agent handles exactly ONE user story, then stops. Fresh context per iteration prevents context exhaustion. Small, atomic, testable increments.

✔

Quality Gates

Every commit must pass quality checks — typecheck, lint, tests. Failed checks trigger retry (up to 3x). Blocked stories are recorded, never left broken.

⇄

Parallel Execution

Independent phases run simultaneously. Dependency analysis identifies which phases can safely overlap. Synchronization points between groups propagate learnings.

🔄

Prompt Repetition

The task is restated at the start and end of every phase and every story. Ensures the model maintains full attention across long contexts and many iterations.

📈

Progress Tracking

Three layers: prd.json flags (state machine), todo.md checkboxes (human view), and progress.txt learnings (agent memory). Full visibility at all times.

🔗

Dependency Mapping

Each phase declares what it produces and consumes. Hard dependencies are blocking, soft dependencies are advisory. The critical path determines minimum execution time.

🧬

Pattern Propagation

Codebase patterns discovered in earlier phases are extracted from progress.txt and injected into later phases' agent context. Ensures consistency across parallel work.

🛠

Error Recovery

3-attempt retry with diagnosis. If still stuck: record blocker in prd.json, reset changes, move to next story. Workflow is fully resumable from any interruption.

📏

Story Right-Sizing

Each story must be completable in one agent pass (one context window). Describable in 2-3 sentences. If it can't, split it. Acceptance criteria must be objectively verifiable.

⏯

Full Resumability

Interrupted mid-execution? Just restart. prd.json tracks which stories passed. progress.txt preserves learnings. Git preserves code. Pick up exactly where you stopped.

📚

Model Tiering

Start with a cost-effective model, auto-escalate to a more capable one after consecutive failures, de-escalate after success. Balances cost and capability across stories of varying difficulty.

🗳

Consensus Verification

Raft-inspired verification built into every phase: after implementation, a quorum of verifier agents independently review the code and cast VOTE:PASS or VOTE:FAIL. Majority required to commit. Catches errors a single agent might miss.

🛡

Iron Laws

Three inviolable rules enforced in every agent: ONE STORY per iteration, NEVER COMMIT BROKEN code, READ PROGRESS.TXT FIRST. Rationalization tables help agents catch self-deceptive shortcuts.

↩

Rollback & Recovery

Snapshot tags before each attempt enable clean rollback. Structured failure reports in failure_log.json give future agents context to try different approaches instead of repeating the same mistake.
